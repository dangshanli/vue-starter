# ★前置环境以及相关配置

# 平台环境

k8s部署平台:

  • master 10.0.108.11

  • node 10.0.108.12

redis statefule容器资源配置：

          resources:
            limits:
              cpu: '4'
              memory: 8Gi
            requests:
              cpu: '4'
              memory: 4Gi

永久持久化位置为nfs服务，地址为：

  nfs:
    path: /data/k8s/redis/kcsp
    server: 10.0.108.11

关闭透明内存大页(THP)，同时设置somaxconn为4096，以满足redis的基本部署需求

预填充数据400万，初始内存占用300M，为后续压测预留一定空间：

# Redis专属配置

    save 600 1
    stop-writes-on-bgsave-error yes
    rdbcompression yes
    rdbchecksum yes
    dbfilename dump_pv.rdb
    dir /data
    aof-use-rdb-preamble no
    maxmemory 7168mb
    maxmemory-policy allkeys-lru

简单说明：

  1. 最低10min持久化一次，持久化错误则关闭写入能力（因为这是服务出现问题的迹象）

  2. rdb文件压缩以及校验和开启

  3. 配置关闭混合持久化模式（5.x以后版本默认打开）

  4. redis最大内存使用限制在7G，超过最大内存启用LRU过期策略

相关部署文件可参见10.0.108.11节点下/home/ranni/appyaml/redis-all.yaml文件。

# ★基准测试以及结果展示

# 测试指令以及相关参数说明

基准测试使用官方提供的benchmark工具，

./redis-benchmark -a 123456 -r 100000 -n 500000 -t set,get,hset
 -l >> /data/pod-inner-513.log &

简单说明：

  • 使用10万随机key做查询，模仿真实情况，每个方法单轮操作50万次
  • 默认50并发线程，关闭pipeline选项，凸显时延影响
  • 测试时长30+min
  • 只针对kcsp项目中常用的set、get、hset几个操作测试
  • 日志写到nfs盘的pod-inner-513.log，后面会提供一个可供查阅的地址
  • 使用默认payload，为3byte

备注：本次测试为进入Pod内部指令测试，并非使用NodePort端口转发的方式。

以上片段截取自记录文件，set、get总体qps在4万+，而hset总体稳在3万+。需要说明一下，这些数值仅仅只是参考，测试方式以及部署的硬件环境都会对最终结果产生巨大影响。以硬件而论，比如pod运行机器的cpu主频、是否支持超频，内存频率、颗粒好坏、是否双通道等皆有影响。笔者之前在10.0.108.1集群上部署，其qps测试结果总体为6万+左右，而当前笔者的集群机器10.0.108.12为长城提供的整机。所以对数据本身要理性看待。

就时延占比来看，get、set操做99%以上会集中在1ms时延，而更加复杂的hset操作99%的操作占比在2ms，总的来说三种操作99.9%的操作都会在4ms以内。
